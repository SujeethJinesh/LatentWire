# ICLR-Grade Evaluation Framing for LatentWire

This memo is the evaluation contract for positive-method LatentWire claims. If a result is meant to support a paper claim, it should survive paired significance, uncertainty reporting, matched-budget controls, contamination checks, and reproducible artifacts.

| **McNemar for paired accuracy** | https://link.springer.com/article/10.1186/1471-2288-13-91 | For paired binary outcomes, compare the discordant pairs; this is the right test family for before/after or baseline-vs-method accuracy on the same examples. | Use a paired test on example-level correctness whenever LatentWire is compared against a baseline on the same items. Report the discordant counts, not just aggregate accuracy deltas. | `n01`, `n10`, paired correctness vector, per-example ids, exact p-value, test variant, task/split, seed. | Unpaired tests and aggregate-only tables can overstate wins; tiny test sets make a few examples swing the result. |
| **Bootstrap methods** | https://statistics.stanford.edu/technical-reports/bootstrap-methods-another-look-jackknife | Resampling estimates uncertainty without strong parametric assumptions. For eval, this is the default way to put a confidence band around a metric or metric delta. | Report 95% bootstrap CIs for LatentWire deltas on the same paired sample set; use stratified or paired bootstrap when the examples are not i.i.d. | sample-level correctness, metric delta, bootstrap seed, resample count, CI low/high, cluster id if grouped data, task size. | Point estimates without uncertainty make small gains look stable when they are not; naive bootstrap can be wrong if grouping or dependence is ignored. |
| **Paired bootstrap in current LLM eval** | https://arxiv.org/abs/2406.06474 | Modern LLM evaluation papers already use paired bootstrapping to compare models and to report 95% CIs. | Match that standard: report paired bootstrap CIs on every headline LatentWire comparison, and mark low-sample pairwise comparisons as underpowered. | paired ratings, rater/model id, CI low/high, bootstrap iterations, per-section metrics, multiple-comparison correction flag. | A result can be numerically higher yet statistically indistinguishable; low n should be called out explicitly. |
| **Test-time cost accounting** | https://arxiv.org/abs/2405.09798 | Accuracy must be reported together with per-query latency and per-example cost; batching, prompt length, and API pricing materially change the interpretation. | Every LatentWire table should report wall-clock, token counts, and any repair/verifier passes. If the method changes compute, the gain must be budget-matched, not implied to be free. | prompt tokens, output tokens, wall-clock ms, p50/p95 latency, batch size, API cost or local FLOPs, repair rounds, verifier calls, transport bytes. | A method can look better only because it spends more test-time compute, more batch efficiency, or more hidden repair passes. |
| **LM Evaluation Harness** | https://github.com/EleutherAI/lm-evaluation-harness | Public prompts, standardized task configs, sample logging, and backend-specific but documented evaluation flows improve comparability and reproducibility. | Use a frozen prompt manifest, frozen task config, logged samples, exact command line, and pinned model revision for every LatentWire benchmark run. | task name/version, prompt hash, doc ids, model revision, decoding args, `output_path`, `log_samples`, cache hits, postprocessing version. | Prompt drift, hidden postprocessing, or backend swaps can create fake improvements that do not reproduce outside one run. |
| **GPT-3 contamination controls** | https://arxiv.org/abs/2005.14165 | Clean evaluation requires removing overlapping or potentially leaked examples from train data and then comparing clean-vs-full performance. | For LatentWire, publish a decontaminated score alongside the raw score and show the removal rule used to define contamination. | overlap hash rule, contamination threshold, cleaned sample ids, removed fraction, raw score, clean score, effect size. | Benchmark leakage can inflate the headline number, especially on popular reasoning and QA sets. |
| **PALOMA evaluation controls** | https://arxiv.org/abs/2312.10523 | Rigorous LM evaluation needs fixed tokenization, max sequence length, random seed, consistent input formatting, and explicit decontamination. | Treat tokenization, sequence length, and input formatting as part of the evaluation definition for LatentWire; do not compare runs that differ in these settings. | tokenization version, max sequence length, random seed, input format hash, decontamination method, corpus source, removed docs. | Apparent gains can come from a different tokenizer, a different packing rule, or a cleaner evaluation slice rather than from the method itself. |
| **Model Cards for Model Reporting** | https://arxiv.org/abs/1810.03993 | Release documentation should cover intended use, performance across conditions, and known limitations instead of only a single score. | LatentWire should ship a model-card-style eval note with benchmark slices, intended use, failure modes, and known non-goals. | intended use, out-of-scope use, benchmark slices, subgroup metrics, failure examples, limitations, calibration notes. | A single average score hides slice regressions and makes a positive-method paper read like a deployment claim. |
| **OpenAI GPT-4o System Card** | https://cdn.openai.com/gpt-4o-system-card.pdf | System cards pair capability numbers with risk areas, mitigation descriptions, and known limitations from red teaming and targeted evals. | For LatentWire, include a system-card-style telemetry block for failure modes, mitigations, and residual risks whenever the method changes routing, repair, or communication policy. | risk category, red-team prompt class, refusal rate, harmful output rate, mitigation applied, residual failure examples, eval protocol. | A method can improve one metric while making safety, refusal behavior, or robustness worse in a way that a single score will not show. |

## LatentWire Evaluation Contract

1. If a claim is about accuracy, report paired significance and a 95% CI on the paired delta.
2. If a claim is about efficiency, report token, byte, latency, and repair costs together.
3. If a claim is about robustness, report cleaned-vs-raw scores and the contamination rule.
4. If a claim is about generality, report slice metrics, not just one benchmark average.
5. If a claim is about reproducibility, ship the exact command, config, prompt manifest, and sample logs.

## Top 5 Evaluation Upgrades For LatentWire

1. Add paired-example significance tests and bootstrap CIs to every main table.
2. Report a full compute ledger: tokens, bytes, wall-clock, repair rounds, and verifier calls.
3. Freeze a public prompt/task manifest and run everything through the same LM-eval-style harness.
4. Publish raw-vs-clean contamination results with explicit overlap rules and removal fractions.
5. Package a model-card/system-card style artifact with slice metrics, failure cases, and residual risks.
