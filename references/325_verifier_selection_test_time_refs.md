# Verifier, Selection, and Test-Time Compute References for LatentWire

Web check: 2026-04-21. This memo collects primary sources that map cleanly onto LatentWire candidate selection: listwise reranking, process and outcome reward models, self-verification, adaptive test-time compute, and confidence-gated sampling.

| Source | Primary links | Why it matters for LatentWire | Concrete LatentWire ablation |
|---|---|---|---|
| **Rank-K: Test-Time Reasoning for Listwise Reranking** | [arXiv](https://arxiv.org/abs/2505.14432) | Treats reranking as a reasoning problem at test time, which is the closest IR analogue to selecting among latent-wire candidates with a model-aware comparator. | Turn each candidate route into a list item and compare listwise ranking against independent pointwise scoring, majority vote, and oracle top-1 selection at fixed candidate count. |
| **Self-Calibrated Listwise Reranking with Large Language Models** | [arXiv](https://arxiv.org/abs/2411.04602) | Shows that listwise scores need calibration across the whole set, not just per-item confidence, which matters when candidate routes are interdependent. | Add a list-view scorer over route sets and calibrate it with point-view features such as route entropy, source-target agreement, and answer-format validity. |
| **Generative Verifiers: Reward Modeling as Next-Token Prediction** | [arXiv](https://arxiv.org/abs/2408.15240) | Makes the verifier itself generative, which is useful if LatentWire wants a lightweight verifier that reasons over candidate telemetry instead of a heavy classifier. | Replace a scalar selector with a small generative verifier over candidate summaries and compare it to a pure classifier on the same route features. |
| **Rewarding Progress: Scaling Automated Process Verifiers for LLM Reasoning** | [arXiv](https://arxiv.org/abs/2410.08146) | Strong evidence that process signals can beat outcome-only ranking when the selector can observe progress instead of only the final answer. | Add a process score that checks whether each candidate route becomes more answer-relevant over time, then compare process-only, outcome-only, and hybrid selection. |
| **The Lessons of Developing Process Reward Models in Mathematical Reasoning** | [arXiv](https://arxiv.org/abs/2501.07301) | Highlights that PRM quality depends on label quality and on how the reward is aggregated, which is exactly the failure mode for latent candidate selection. | Evaluate whether step-level route checks outperform final-answer-only selection, and log where PRM-style scores disagree with outcome correctness. |
| **Large Language Models Can Self-Correct with Key Condition Verification** | [arXiv](https://arxiv.org/abs/2405.14092) | Provides a simple self-verification loop: mask a key condition, ask the model to reconstruct it, and use that as a check on the answer. | For each candidate, generate one or two key-condition checks from the prompt and score candidates by whether they preserve those conditions. |
| **Scaling LLM Test-Time Compute Optimally can be More Effective than Scaling Model Parameters** | [arXiv](https://arxiv.org/abs/2408.03314) | Directly supports compute allocation by difficulty instead of spending the same candidate budget on every prompt. | Route only hard examples through multi-candidate latent-wire selection using a cheap uncertainty trigger, and keep easy examples on single-pass decoding. |
| **Confidence Improves Self-Consistency in LLMs** | [arXiv](https://arxiv.org/abs/2502.06233) | Weighted voting by model confidence can outperform plain self-consistency and reduce the number of samples needed. | Replace majority vote with confidence-weighted selection over candidate routes using route confidence, answer confidence, and margin between top candidates. |
| **Learning When to Sample: Confidence-Aware Self-Consistency for Efficient LLM Chain-of-Thought Reasoning** | [arXiv](https://arxiv.org/abs/2603.08999) | Explicitly trains a gate that decides when single-sample decoding is enough and when extra samples are worth paying for. | Train a selector that decides between 1, 3, and 5 latent-wire candidates from cheap features such as entropy, disagreement, and answer extraction stability. |

## Highest-yield LatentWire Ablations

1. **Listwise selector vs pointwise selector.** Score the full candidate set jointly rather than independently and test whether this recovers correct routes that pointwise ranking misses.
2. **Process vs outcome vs hybrid reward.** Compare final-answer scoring, step-progress scoring, and a linear mixture to determine whether candidate quality is being missed because the selector is too outcome-only.
3. **Confidence-gated sampling.** Only spend extra latent-wire candidates on uncertain prompts and match average latency to a fixed-budget baseline.
4. **Self-verification on candidate summaries.** Ask the model to verify key conditions implied by the prompt and use that as a candidate filter before decoding.
5. **Calibrated weighted voting.** Replace majority vote with confidence-weighted aggregation and measure whether the gain comes from better selection or just from more samples.
