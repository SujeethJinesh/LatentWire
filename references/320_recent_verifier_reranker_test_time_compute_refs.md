# Recent Verifier, Reranker, and Test-Time Compute References for LatentWire

Primary-source memo for improving LatentWire's stochastic route candidate selection. The recurring pattern in the recent literature is: generate multiple candidates, score them with a verifier that uses either process signals or confidence signals, and only spend extra compute when the first pass is uncertain.

| Source | Primary links | Mechanism worth borrowing | Concrete LatentWire ablation |
|---|---|---|---|
| **Scaling LLM Test-Time Compute Optimally can be More Effective than Scaling Model Parameters** | [paper](https://arxiv.org/abs/2408.03314) | Compute should be allocated adaptively per prompt. The paper compares search against process-based verifiers and adaptive distribution updates, and shows compute-optimal allocation beats naive best-of-N. | Gate stochastic route expansion by target-alone confidence or entropy. Compare fixed-N routing vs difficulty-gated N under matched bytes and latency. |
| **Generative Verifiers: Reward Modeling as Next-Token Prediction** | [paper](https://arxiv.org/abs/2408.15240), [data/code](https://github.com/genrm-star/genrm-critiques) | A verifier can be trained as a generative next-token predictor, so the same model family can generate reasoning and verify candidates. Majority voting over verifier rationales is a first-class test-time knob. | Replace a scalar route score with a generative verifier prompt over route telemetry: route trace, target answer, source-target agreement, and a binary accept/reject rationale. Compare discriminative vs generative scoring. |
| **Rewarding Progress: Scaling Automated Process Verifiers for LLM Reasoning** | [paper](https://arxiv.org/abs/2410.08146) | Step-level reward should measure progress, not just final correctness. Process Advantage Verifiers are explicitly designed to improve search and RL by scoring whether a step increases the chance of a correct final answer. | Score each route with a stepwise progress proxy instead of only final answer quality. Useful features: intermediate consistency, answer deltas, route entropy drop, and source-target disagreement reduction. |
| **The Lessons of Developing Process Reward Models in Mathematical Reasoning** | [paper](https://arxiv.org/abs/2501.07301) | PRM quality depends heavily on how labels are synthesized and how BoN is evaluated. The paper argues that MC-estimated labels are often weaker than LLM-as-judge or human labels, and that BoN can be biased toward outcome-only behavior. | Train or evaluate LatentWire verifiers with three label sources: MC self-labels, LLM-judge labels, and a small gold subset. Check whether route selection changes when the verifier sees process labels rather than outcome labels. |
| **SETS: Leveraging Self-Verification and Self-Correction for Improved Test-Time Scaling** | [paper](https://arxiv.org/abs/2501.19306) | Sampling, self-verification, and self-correction are fused into one test-time loop. The method is aimed at getting better scaling laws than repeated sampling or reward-only selection. | Add a second-stage repair pass only for the top uncertain route candidates. Compare rerank-only vs rerank-plus-repair at the same wall-clock budget. |
| **Rank1 / Rank-K / REARANK** | [Rank1 paper](https://arxiv.org/abs/2502.18418), [Rank1 code](https://github.com/orionw/rank1), [Rank-K paper](https://arxiv.org/abs/2505.14432), [Rank-K code](https://github.com/hltcoe/rank-k), [REARANK paper](https://arxiv.org/abs/2505.20046), [REARANK code](https://github.com/lezhang7/Rearank) | These are the clearest listwise reranking analogs. All three turn candidate ranking into a reasoning task with explicit test-time compute, which maps cleanly onto selecting among stochastic routes. | Treat route candidates as a listwise ranking problem. Compare pointwise score, pairwise tournament, listwise reasoning, and RL-trained reranker selection on the same route set. |
| **Budget-aware Test-time Scaling via Discriminative Verification** | [paper](https://arxiv.org/abs/2510.14913), [code](https://github.com/wang-research-lab/verification) | Discriminative verifiers can be weak in isolation but strong when combined with self-consistency under a fixed compute budget. The paper explicitly argues for a hybrid budget-aware verifier stack. | Add a cheap discriminative filter before any expensive generative verifier. Compare all-generative vs hybrid discriminative-plus-self-consistency vs self-consistency-only. |
| **Reliability-Aware Adaptive Self-Consistency for Efficient Sampling in LLM Reasoning** | [paper](https://arxiv.org/abs/2601.02970) | Adaptive self-consistency should stop based on evidence sufficiency, not raw sample count. The method uses response-level confidence to decide whether to keep sampling. | Start with one route, then expand only when confidence is insufficient. Compare fixed 4-route or 8-route self-consistency against confidence-aware stopping. |
| **ReVISE: Learning to Refine at Test-Time via Intrinsic Self-Verification** | [paper](https://arxiv.org/abs/2502.14565) | Self-verification and correction are trained jointly, and confidence-aware decoding helps the model refine its own reasoning at inference time. | After selecting a top route, allow one targeted repair step instead of full resampling. Measure whether repair recovers errors that reranking misses. |
| **Putting the Value Back in RL: Better Test-Time Scaling by Unifying LLM Reasoners With Verifiers** | [paper](https://arxiv.org/abs/2505.04842) | RL^V keeps a value/verifier signal alive during RL so test-time scaling can use verification without adding a separate heavyweight verifier model. It also supports parallel and sequential test-time compute. | If LatentWire ever learns a route scorer, keep it as a live value signal rather than a post-hoc classifier. Compare rerank-only selection against a jointly trained reasoner+verifier head. |
| **V1: Unifying Generation and Self-Verification for Parallel Reasoners** | [paper](https://arxiv.org/abs/2603.04304) | Parallel candidates are ranked by an uncertainty-guided tournament, and the verifier is trained jointly with generation so it tracks the generator distribution. | Replace flat top-k scoring with pairwise route tournaments when route uncertainty is high. Test whether tournament ranking is more stable than a single scalar route score. |
| **Large Language Models have Intrinsic Self-Correction Ability** | [paper](https://arxiv.org/abs/2406.15673) | A plain model can sometimes revise itself successfully if prompts are fair and decoding is deterministic. This is a useful low-cost correction baseline before adding external verification. | Use a zero-temperature revise pass after the chosen route. Compare revise-after-rerank against rerank-only and against a second stochastic sample. |

## What this means for LatentWire

- Route selection should probably be treated as a budgeted ranking problem, not as a pure sampling problem.
- The strongest recurring signal is not raw answer vote count but route-level evidence: confidence, progress, or self-verification trace.
- The best practical stack is likely hybrid: cheap discriminative filter first, then a generative verifier or repair pass only for uncertain candidates.
- Process-style signals matter because they can separate "looks right" from "moves toward correctness", which is exactly the failure mode of plain route majority vote.
- Any verifier win must be checked against hackability and false negatives, especially if the verifier is also trained on model-generated traces.

## Highest-priority LatentWire ablations

- Fixed-N route sampling vs confidence-gated sampling at matched bytes and wall-clock.
- Outcome-only route score vs process-progress score vs generative verifier score.
- Pointwise selection vs pairwise tournament vs listwise reranking over the same candidate routes.
- Self-consistency-only vs discriminative-filter-plus-self-consistency hybrid.
- Rerank-only vs rerank-plus-targeted-repair on the chosen route.
